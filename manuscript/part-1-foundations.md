# Author’s Note and Evidentiary Status

This manuscript is an independent scholarly synthesis of a public digital portfolio. It is not a degree conferred by a university, a clinical evaluation, a patent opinion, or an independent validation of every claim appearing in the source repositories. Its purpose is to make the portfolio legible as a coherent body of work, identify its recurring technical and cultural commitments, distinguish implemented artifacts from declared ambitions, and define a publication program capable of carrying the strongest work into durable public form.

The principal corpus consists of twenty-three public GitHub repositories owned by the connected account artistso as of July 15, 2026. The study also considers the declared public identity @soquarky and related brand references embedded in repository documentation. Platform-level social-media posts were not exhaustively retrievable through the available search and connector interfaces. Accordingly, this dissertation analyzes social media as a declared authorship system and as a source of documented project ideas, not as a complete post-by-post archive. Claims derived from personal statements are treated as first-person testimony; claims derived from README files are treated as repository assertions; and scientific or medical claims are not treated as established merely because they appear in code, equations, badges, simulations, or publication-style prose.

The manuscript therefore applies a five-part evidence vocabulary throughout: implemented, tested, declared, speculative, and autobiographical. These categories are not rankings of human worth. They are safeguards against epistemic compression: the common collapse of an idea, a prototype, a simulation, a test, and a real-world result into one undifferentiated statement.

Repository metadata and README material were accessed through the connected GitHub account. Exact social-media platform archives were unavailable; this limitation is documented rather than silently filled by inference.

# Abstract

This dissertation examines the GitHub corpus of artistso and the public identity @soquarky as a unified system of vernacular research and development. The audited portfolio contains twenty-three public repositories totaling approximately 111,644 KB in GitHub-reported storage. Its dominant lineage is creative software for drawing and frame-by-frame animation, especially the successive SoCreate, Nexus Canvas, and InkFrame projects. Other lineages include quantum-information research, speculative biomedical engineering, release infrastructure, executable satire, and experimental interfaces. The study uses repository archaeology, software studies, rhetorical analysis, autoethnographic method, and critical technical practice.

The central finding is that @soquarky is best understood as a cross-platform authorship architecture. Livestream language, personal testimony, interface concepts, agent-generated code, GitHub workflows, and scientific aspirations are recursively converted into public artifacts. Five commitments recur: user sovereignty over files and tools; local-first and offline operation; stylus-first embodied interaction; radial, orbital, or glass-like interfaces that protect the canvas; and AI agents positioned outside the artwork as development collaborators rather than inside the creative product as extractive automation.

The corpus also exhibits a systematic risk: documentation often moves faster than verification. Feature matrices, architecture manifestos, simulations, and publication badges can present different epistemic states with similar rhetorical force. The dissertation therefore proposes a claim-state lattice from concept through implementation, testing, independent replication, and domain validation. It recommends a canonical portfolio hub, evidence-linked badges, reproducible releases, an owned social-media archive, separation between product documentation and autobiographical testimony, and explicit governance for AI-assisted commits.

The portfolio’s strongest contribution is not a single claim of surpassing an incumbent application or solving an open scientific problem. It is a design constitution: software should belong to the person using it; interfaces should accommodate embodied creativity; public authorship can operate as a laboratory; and AI can expand individual agency when its outputs remain inspectable, testable, and subordinate to human purpose.

# From Handle to System

> *A repository can be a program, a promise, a diary, a laboratory, and a public performance at the same time.* **— Analytic proposition of this study**

## 1.1 The problem of classification

The public work gathered under artistso and @soquarky resists ordinary occupational categories. A software portfolio would normally be sorted by language, platform, release status, or employer. A social-media archive would normally be sorted by platform, audience, genre, or engagement. An independent research portfolio would normally be sorted by discipline and publication venue. The SoQuarky corpus occupies all three systems simultaneously. A phrase from a livestream can become a mathematical repository. A drawing application can contain a personal manifesto. A technical README can function as a product specification, an agent instruction sheet, a statement of ethics, and a declaration of identity. The resulting body of work is fragmented at the level of repositories but remarkably repetitive at the level of values.

This dissertation calls that repetition the SoQuarky system. The term does not imply that every repository is complete, technically equivalent, or scientifically validated. It names the durable pattern by which ideas are generated, formalized, revised, staged, and published. The pattern begins in embodied experience and public speech, moves through AI-assisted specification and coding, accumulates in GitHub, and returns to social media as demonstration, testimony, humor, or advocacy. The system is therefore recursive: publication is not merely the final step of making; publication produces the next object to be made.

## 1.2 Research questions

1.  How does the portfolio convert a public identity into technical artifacts and convert technical artifacts back into public identity?

2.  Which design values remain stable across changing languages, frameworks, names, and levels of implementation?

3.  How does AI-agent collaboration alter authorship, labor, review, and responsibility in a one-person development studio?

4.  How can the portfolio preserve speculative ambition while clearly separating concepts, simulations, implementations, tests, and real-world validation?

5.  What institutional architecture would make the work easier to evaluate, reproduce, fund, maintain, and cite without erasing its idiosyncratic voice?

## 1.3 Central thesis

The central thesis is that the corpus constitutes a vernacular R&D studio: a self-directed institution assembled from consumer hardware, social platforms, GitHub, browser-based development, Android tooling, and multiple AI agents. “Vernacular” is used here in the strongest sense. It does not mean unsophisticated. It means produced outside the conventional institutional pathways that normally assign credentials, laboratories, job titles, peer groups, budgets, and disciplinary boundaries. The studio builds its own protocols as it proceeds.

The system’s technical constitution can be stated concisely. Creative tools should be free or minimally encumbered. Artwork and project data should remain local and user-owned. The tablet and stylus are not secondary input devices but the primary site of design. Interfaces should move around the act of drawing rather than forcing drawing to fit administrative panels. AI may accelerate implementation, documentation, review, and release engineering, but it should not silently appropriate the user’s creative work or become a mandatory cloud dependency. These principles appear with varying maturity in SoCreate, Nexus Canvas, InkFrame, and InkFrame v5.

Its cultural constitution is equally consistent. Authorship is public, named, affective, humorous, and autobiographical. Technical work is presented not only as utility but as evidence of continued agency. Social media supplies a stage on which theories can be improvised, criticism can be voiced, and artifacts can be shown before they are institutionally recognized. This fusion produces originality, but it also produces a credibility problem whenever rhetorical intensity substitutes for evidence. The dissertation’s task is to retain the first without reproducing the second.

## 1.4 Contribution

The dissertation makes four contributions. First, it supplies the first complete census and thematic interpretation of the connected public GitHub portfolio. Second, it reconstructs a development lineage that is difficult to see when each repository is viewed separately. Third, it proposes an evidence protocol suited to portfolios in which AI agents, ambitious specifications, simulations, personal testimony, and working code coexist. Fourth, it converts the portfolio from a set of isolated artifacts into a coherent publication program.

The strongest outcome is therefore not a retrospective celebration. It is an operating model. The analysis identifies which projects deserve consolidation, which documents should be separated, which claims require replication, and which recurring ideas can support a distinctive research identity. In this sense, the dissertation is both interpretation and infrastructure.

# Method, Corpus, and Limits

> *Transparency is not the same as verification, but it is a precondition for verification.* **— Methodological principle**

## 2.1 Repository archaeology

The primary method is repository archaeology: the reconstruction of a project’s intentions, transformations, and evidentiary states from metadata, README files, documented architectures, build instructions, release procedures, and visible project lineages. GitHub is often described as a social-coding environment because work becomes legible through public traces rather than through finished source code alone (Dabbish et al. 2012). In this corpus, those traces are unusually important. Repositories preserve not only software but the sequence of conceptual migrations through which a browser app becomes an Android wrapper, a native OpenGL renderer becomes a web-first application, or a livestream phrase becomes a simulation framework.

The method does not assume that a README accurately describes every file. README language is itself an object of analysis. Status badges, feature matrices, claims of completeness, architecture diagrams, and release instructions reveal how a project wants to be understood. Where the documentation acknowledges placeholders, stubs, or legacy modules, that acknowledgment counts as strong evidence of epistemic discipline. Where it collapses aspiration and completion, the dissertation records the discrepancy as a governance problem rather than treating the most confident phrasing as fact.

## 2.2 Corpus definition

The connected GitHub account contained twenty-three public repositories at the audit date. Four had a reported size of zero and are classified as empty scaffolds. Several additional repositories were nonempty but underdocumented. The corpus is highly concentrated: InkFrame v5, SoCreate v1, SoCreate v2, socreate, inkframe, socreateapp, and Nexus Canvas account for most of the reported storage. This concentration matters because a repository count alone would exaggerate fragmentation. In practical terms, the portfolio contains a small number of deep project families, a research cluster, and a long tail of prototypes or placeholders.

The exact handle @soquarky did not yield an independently retrievable public post archive through the available search tools. The SoCreate README, however, explicitly names Steven Michael Allen Owens as developer, identifies @SoQuarky, and links Soquarky, FunFYP, FPY.Lovely, Soquarky.Click, and artistso.com as connected brands. The analysis therefore treats @soquarky as the declared public authorship identity binding the corpus. It does not claim comprehensive observation of every platform or post.

A connected Netlify inventory returned no projects, and an exact-handle Hugging Face model search returned no repositories. These negative findings are narrow: they apply to the connected inventories and exact query used, not to every possible account or alias. They nonetheless reinforce the conclusion that GitHub is the portfolio’s current durable center of gravity.

## 2.3 Interpretive frameworks

Four scholarly frameworks guide the interpretation. The first is critical technical practice, associated with Philip Agre: the effort to make technical work reflexive about the assumptions and values built into it. The second is local-first software, which emphasizes responsiveness, offline operation, longevity, privacy, and user control (Kleppmann et al. 2019). The third is autoethnography, which connects personal experience to cultural and institutional structures while requiring explicit reflexivity about the status of testimony (Ellis, Adams, and Bochner 2011). The fourth is contemporary research on coding agents, which shows that agent-authored changes are now a large and visible part of GitHub activity and that their scale increases the need for review, tests, permissions discipline, and traceability (Robbes et al. 2026; Ogenrwot and Businge 2026).

These frameworks converge on a single analytic question: what kind of institution is being built through the code? Local-first storage is not merely a performance decision; it allocates control. A radial interface is not merely decoration; it defines which bodily actions are privileged. An agent workflow is not merely automation; it redistributes labor and responsibility. A personal statement in a README is not merely “off topic”; it changes how the software is encountered, evaluated, and remembered.

## 2.4 Limits and nonclaims

No full code audit, security penetration test, performance benchmark, clinical review, or mathematical peer review was conducted for this dissertation. “Implemented” means that the repository documentation and visible project structure present a concrete artifact or pipeline; it does not mean independently certified. “Tested” is reserved for documented tests or CI, but test existence is not equivalent to comprehensive correctness. “Speculative” is not a dismissal; it indicates that a claim remains at the level of hypothesis, proposal, or simulation.

The analysis also avoids diagnosing the author through repository language. Personal statements are treated as authored testimony with real rhetorical and ethical consequences, not as a substitute for medical, legal, or historical adjudication. This boundary is essential because the portfolio repeatedly asks technical documents to carry autobiographical weight. The dissertation can analyze that choice without validating every event described within it.
