# Risk, Governance, and Institutional Legibility

> *A project becomes fundable when another person can tell what is true, what is next, and what help is needed.* **— Portfolio-governance principle**

## 11.1 The principal risks

6.  Fragmentation: multiple repositories compete to represent the same product lineage.

7.  Status inflation: feature matrices and badges imply more completion or validation than the evidence supports.

8.  Genre collision: product documentation, testimony, satire, and research claims occupy the same rhetorical surface.

9.  Agentic opacity: large AI-generated changes may be accepted without a durable review record.

10. Security exposure: tokens, keystores, signing instructions, and third-party actions create supply-chain risk.

11. Maintenance overload: one author cannot sustainably maintain every prototype as an active product.

12. Platform fragility: social-media history is not reliably preserved or searchable.

## 11.2 Canonicalization

The first governance act is to choose canonical homes. InkFrame v5 should be the canonical creative-software implementation. Earlier SoCreate, InkFrame, Nexus Canvas, Draw, and quantum-named art prototypes should be marked as legacy, research, architecture, or archive, with prominent links to the flagship. Whitepapers should be the canonical research-method and validator repository. The quantum-bounce project should remain a focused paper repository. SynapShield should be reframed as a computational concept study unless and until external validation changes its state. Gutty should be explicitly categorized as computational satire.

Canonicalization does not destroy history. It turns history into a navigable lineage. Each legacy repository can preserve its code while displaying a status banner: superseded, archived, concept only, or active research.

## 11.3 Evidence badges

The portfolio already uses familiar badges for CI, version, license, and publication. It should add evidence badges with strict definitions: CONCEPT, IMPLEMENTED, TESTED, REPRODUCED, and DOMAIN-VALIDATED. These badges should link to evidence, not merely decorate a README. A TESTED badge links to a workflow and test specification. A REPRODUCED badge links to an external report. A publication badge links to the actual record and status.

This system would allow ambitious projects to remain visible without borrowing credibility from later stages. It would also make negative results publishable: a concept can progress to a tested obstruction even when the original hypothesis fails.

## 11.4 Security and release engineering

The build template and InkFrame CLI demonstrate strong attention to GitHub Actions, signing, artifacts, and immutable action SHAs. These practices should be centralized. Secrets should be rotated if ever exposed. Permanent signing keys should never be generated or transported through conversational text. Release workflows should be fail-closed, branch-protected, and reproducible from tagged source. Each binary should carry checksums and a provenance statement.

Agentic systems increase the importance of these controls because an agent can execute instructions at speed. Dependency allowlists, network restrictions, protected environments, and manual release approval are not bureaucratic obstacles; they are the boundary that permits experimentation without losing the repository.

## 11.5 Institutional legibility without domestication

Institutional legibility often rewards blandness, but the answer is not to remove cosmic metaphors, humor, autobiography, or rose-quartz interface language. It is to pair them with conventional evidence surfaces: a funder can enjoy the Glass Horizon and still find a test plan; a researcher can encounter the Dangle Axiom and immediately see that it is satire; a clinician can inspect SynapShield and see the boundary between simulation and proposed intervention. Legibility is translation, not surrender. Identity becomes stronger when readers need not guess which statements are literal.

CHAPTER 12

# A Coherent Research and Publication Program

> *The next project is not another repository. It is the architecture that makes the repositories cumulative.* **— Strategic conclusion**

## 12.1 The studio model

The recommended institutional identity is SoQuarky Studio: an independent, public-interest creative technology and research studio. “Studio” is preferable to “laboratory” as the umbrella because it can contain software, animation, writing, comedy, interface experiments, and formal research without implying that every output is experimentally validated. Individual divisions can then use more specific language.

- InkFrame Studio: flagship free drawing and animation software.

- Glass Horizon Research: radial, orbital, and embodied interface studies.

- SoQuarky Papers: quantum information, formal validators, and reproducible notes.

- Public Systems Commentary: first-person essays and humane-technology proposals.

- Computational Comedy Lab: Gutty, absurdist simulations, and executable satire.

- Archive: superseded repositories, design records, transcripts, and project histories.

## 12.2 The flagship research question

The most defensible cross-disciplinary research question is: How can local-first, stylus-first creative software use radial and orbital interfaces to reduce administrative friction while preserving user ownership and transparent AI-assisted development? This question is original enough to distinguish the portfolio and concrete enough to study. It unifies the actual software, the interface theory, the ethics, and the agent workflow.

A research program can produce both artifacts and papers: InkFrame releases, controlled interaction studies, archive-schema documentation, privacy audits, and analyses of agent-assisted one-person development. Each output strengthens the others.

## 12.3 Twelve-month program

13. Create a canonical portfolio repository and static site with the complete census, status labels, and links.

14. Freeze legacy creative repositories except for archival READMEs pointing to InkFrame v5.

15. Publish InkFrame v1.0 criteria: supported devices, archive stability, export reliability, privacy audit, and accessibility baseline.

16. Implement the evidence-badge schema across research repositories.

17. Separate personal statements and public-interest essays from product READMEs while preserving them in linked, versioned documents.

18. Run a small radial-interface usability study with preregistered tasks and local-only telemetry.

19. Release the quantum recoverability validator as a narrow, tested package before extending gravitational interpretation.

20. Reframe SynapShield documentation as a computational hypothesis and add sensitivity, uncertainty, and failure analysis.

21. Create an owned social-media archive with transcripts, project links, and provenance.

22. Adopt branch protection, review templates, agent provenance, dependency review, and release approvals across active repositories.

## 12.4 What should not be done

The portfolio should not launch another full creative application under a new name before canonicalizing the current line. It should not use publication badges for submissions that cannot be independently located. It should not call simulation output clinical validation. It should not merge autobiographical testimony into every technical entry point. It should not grant broad repository or signing permissions to agents for convenience. It should not delete the history that explains how the flagship emerged.

Restraint is now a form of acceleration. Every new repository increases the cost of explanation. The next phase should deepen evidence, release quality, and audience comprehension.

## 12.5 Long-range significance

The SoQuarky system offers a view of authorship that is likely to become more common: one person directing a shifting ecology of models, repositories, workflows, devices, and publics. The decisive question will not be whether the code was typed by a human alone. It will be whether the resulting institution preserves human purpose, user rights, inspectable evidence, and responsibility.

This portfolio is unusually suited to answer that question because it already contains the contradiction in explicit form: AI builds the environment; the environment protects the human mark. The dissertation’s final recommendation is to make that contradiction the studio’s defining research contribution.
